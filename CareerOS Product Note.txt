WHAT IS CAREEROS
--------------------------------------------------------------------------------

CareerOS is an AI-powered career coaching and job automation platform built
exclusively for the Indian job market. It is a SaaS web application hosted on
Streamlit Cloud, backed by a Python/Playwright automation engine that runs
locally on the user's Windows PC.

The product is opinionated: it understands how Naukri's ranking algorithm works,
how RChilli (Naukri's ATS parser) reads resumes, how Indian recruiters evaluate
profiles in the first 6 seconds, and how the gap between what candidates write
and what gets them callbacks actually looks in practice.

The target user is a mid-career Indian professional (5-12 years experience)
who knows their work is good but doesn't know how to present it, which role
to target, or how to make the system work in their favour.

SECTION 1 — AUTHENTICATION & IDENTITY
--------------------------------------------------------------------------------

Method: Google OAuth 2.0 via Streamlit's native st.login() / st.user API.
Requires Streamlit 1.41 or higher with [auth] section configured in
Streamlit Cloud secrets.

Flow:
  1. User visits the app URL.
  2. If not logged in: a centered landing page appears with a
     "Sign in with Google" button.
  3. User clicks → standard Google OAuth consent screen.
  4. On approval: Streamlit receives the Google ID token.
  5. User identity is extracted from st.user:
        - st.user.email    (primary identity key — used throughout the app)
        - st.user.name     (display name)
        - st.user.picture  (avatar URL)
  6. User is dropped directly into the CareerOS dashboard.

User isolation:
  Every user's data is stored in a directory keyed by MD5 hash of their
  lowercase email. So user saikatpatra64@gmail.com → hash key → directory.
  No two users' data can collide. No database login required.

Sign out:
  Available from the dashboard via a "Sign out" button that calls st.logout().

SECTION 2 — USER ONBOARDING (SETUP PAGE)
--------------------------------------------------------------------------------

Page: 0_Setup.py
URL slug: /Setup

One-time setup page. Can be revisited to update preferences at any time.
Collects three categories of data:

2.1 NOTIFICATION SETUP (ntfy)

  CareerOS auto-generates a unique, non-guessable notification topic for
  each user derived from their email hash. Format: "careeros-{10char_hash}".

  User is instructed to:
    - Install the free ntfy app on their phone (iOS or Android)
    - Subscribe to their unique topic code (shown on screen)
    - Click "Test Notification" to verify the connection

  A test push is fired from the web app to ntfy.sh to confirm it works.
  No account required. No credit card. Completely free.

2.2 NAUKRI CREDENTIALS

  The user enters their Naukri registered email and password.

  Security model:
    - Password is encrypted using AES-256 Fernet symmetric encryption
    - Encryption key (FERNET_KEY) is stored in Streamlit Cloud secrets
      (never on the user's machine)
    - The encrypted password (naukri_pass_enc) is stored in the user's
      profile JSON file on the server
    - The plaintext password is never written to disk at any point
    - When the local automation runner needs to log in, it decrypts the
      password at runtime using the same Fernet key read from config.json

2.3 JOB PREFERENCES

  - Preferred cities (multi-select): Bengaluru, Pune, Mumbai, Hyderabad,
    Chennai, Noida, Gurgaon, Kolkata, Remote, Any
  - Minimum salary in LPA (numeric input)
  - Work mode preference: Any / Work from office / Hybrid / Remote

2.4 CRM INTEGRATION (MAKE.COM)

  On save, if MAKE_CRM_WEBHOOK is configured in Streamlit secrets, the
  setup data (name, email, phone, notification pref, location, salary_min)
  is posted to a Make.com webhook for CRM and marketing use.
  This is non-blocking — if the webhook fails, setup still saves.

All setup data is saved to profile.json in the user's data directory.

SECTION 3 — RESUME BUILDER
--------------------------------------------------------------------------------

Page: 1_Resume_Builder.py
URL slug: /Resume_Builder

This is the core of the product. A multi-turn AI conversation that extracts
the user's career story and generates a professional, ATS-ready resume.

3.1 CONVERSATION ENGINE

  Powered by: Claude Sonnet 4.6 (claude-sonnet-4-6) via Anthropic API
  Model used in probing: max_tokens=1024
  Model used in generation: max_tokens=8192

  The session is managed by ResumeBuilderSession class in
  modules/resume/session.py.

  Flow:
    - A system prompt primes Claude with deep knowledge of Indian job market,
      Naukri algorithm, RChilli ATS parsing, salary benchmarks, and
      experience repositioning
    - Claude opens with a bilingual welcome and starts probing
    - User responds in Hindi, English, or Hinglish — Claude matches the
      user's language automatically (Unicode Devanagari detection)
    - Minimum 6 exchanges before the "Generate Resume" button unlocks
      (configurable via MIN_PROBE_ROUNDS in config.py)
    - Claude signals readiness via a hidden [READY_TO_GENERATE] marker
    - User can continue chatting beyond 6 rounds for more detail

  Session persistence:
    - Chat history is saved to resume_chat.json after each exchange
    - If user returns later, they can "Continue from last session" or
      "Start fresh"
    - restore() classmethod reconstructs the session from saved history

3.2 EXPERIENCE REPOSITIONING INTELLIGENCE (built into the AI prompt)

  The system prompt trains Claude to never accept operational language.
  It repositions every piece of experience:

  Example transformations baked in:
    - "Main call center mein tha" → managed 200+ daily customer interactions,
      identified escalation patterns, reduced repeat complaints by 18%
    - "Main Excel reports banata tha" → Designed MIS dashboards for senior
      leadership, translated transactional data into business insights
    - "Main teams ke beech coordinate karta tha" → Orchestrated cross-
      functional dependencies across product, operations, and delivery teams
    - "Main requirements gather karta tha" → Conducted JAD workshops and
      stakeholder interviews, produced BRDs signed off by 3 teams
    - "Main QA tester tha" → Designed acceptance criteria and traceability
      matrices, reduced post-go-live defects by 35%

  The AI helps users quantify where they have no numbers ("Roughly kitna
  time save hua?").

  Banned clichés the AI is trained to avoid:
    - "Result-oriented professional"
    - "Team player" / "go-getter"
    - "Hard-working and dedicated"
    - "Excellent communication skills" (without evidence)

3.3 ROLE SUGGESTION (generated alongside the resume)

  Claude analyses the user's signals and outputs a role recommendation:
    - Primary recommended role (title)
    - Confidence level: High or Medium
    - Reasoning (2-3 sentences, specific to what the user said)
    - Alternate roles (2-3 options)
    - Domain mismatch warning (if what they want ≠ what they fit)
    - Gap assessment (what's missing to reach their target role)
    - Salary band in India at their experience level (2025-26 market rates)

3.4 RESUME DATA STRUCTURE (JSON)

  Generated resume is stored as structured JSON with these keys:
    - name, target_title, domain_family (enterprise_IT / product / BFSI etc.)
    - summary (professional summary paragraph)
    - ats_keywords (list of 10-20 ATS keywords)
    - skills (dict by category: Technical, Tools, Domain, Soft)
    - experience (list of jobs, each with: company, title, period, bullets)
    - education (list of degrees)
    - certifications (list)
    - role_suggestion (the role recommendation block)

3.5 WORD RESUME EXPORT

  Powered by: python-docx (modules/resume/word_export.py)

  Output: ATS-friendly single-column Word (.docx) file.

  Key design decisions:
    - Single column layout (multi-column fails RChilli parsing on Naukri)
    - Clean formatting with proper heading styles for ATS parsing
    - Filename format: {Name}_{TargetRole}_Resume.docx

  User clicks "Build Word Document" → downloads via browser.
  Success message: "Upload directly to Naukri or attach to job applications."

3.6 RESUME VERSIONING

  Last 5 generated resume versions are stored in resume_history.json.
  Current version is always available from generated_resume.json.

SECTION 5 — ROLE CLARITY TOOL
--------------------------------------------------------------------------------

Page: 3_Role_Clarity.py
URL slug: /Role_Clarity

Standalone tool that doesn't require a built resume.
Use case: User is confused about what role to target.

5 quick questions (form-based, not chat):
  1. Describe your day-to-day work (what you ACTUALLY do most)
  2. What colleagues/manager come to YOU for help with
  3. What part of your work makes time fly (the enjoyable part)
  4. Total years of experience
  5. What role are you targeting (optional)

AI Analysis:
  Claude processes the answers and returns:
    - Primary recommended role
    - Confidence level (High / Medium)
    - Reasoning (specific to what they described)
    - Alternate roles (2 options)
    - Honest advice (if their target differs from recommendation — with
      specific bridge steps, not generic encouragement)
    - Salary band in India for the role at their experience level
    - 3 Next Steps to move toward the recommended role

The "honest advice" section is intentionally direct — if someone says they
want to be a Product Manager but their answers indicate BA-level skills,
CareerOS will say so and explain the specific gaps to bridge.

SECTION 6 — ATS SCORE CHECKER
--------------------------------------------------------------------------------

Page: 4_ATS_Checker.py
URL slug: /ATS_Checker

Requires: Resume must be built first.
AI Model: Claude Sonnet 4.6

Use case: Before applying to any specific job, user pastes the JD to see
how well their resume matches it and exactly what to fix.

Input: Full job description (pasted as text)

6.1 NAUKRI AI RELEVANCE SCORE SIMULATION

  CareerOS simulates the exact 6-parameter scoring system Naukri shows
  to recruiters (High / Medium / Low Match). Output for each:
    - Skills match: candidate skills vs JD required skills (0-100 score)
    - Designation match: title(s) vs job title (0-100)
    - Experience match: years vs JD experience band (0-100)
    - Salary match: expected CTC vs JD budget (0-100)
    - Location match: candidate city vs job location (0-100)
    - Education match: degree vs JD qualification (0-100)

  Displayed as coloured progress bars (green/amber/red based on score).

6.2 OVERALL SCORE AND VERDICT

  Scoring bands:
    - 80-100: Strong Apply
    - 60-79: Apply with Tweaks
    - 40-59: Weak Match — Tailor First
    - 0-39: Skip This Job

  Plus: One-liner biggest risk ("The single biggest reason this resume
  might get rejected — max 15 words"). Brutally honest.

6.3 KEYWORD ANALYSIS

  Keywords Found: JD keywords present in the resume (shown as green pills)
  Keywords Missing: JD keywords absent from resume (shown as red pills)
  User should add missing keywords before applying.

6.4 PRIORITISED RECOMMENDATIONS

  3 specific, actionable recommendations (not generic advice):
    - High Priority (Red): Must fix before applying
    - Medium Priority (Amber): Significant improvement
    - Low Priority (Green): Nice to have

  Each recommendation specifies WHAT to change AND WHERE
  (Resume headline / Skills section / Experience bullet / Profile summary).

6.5 OPTIMISED HEADLINE SUGGESTION

  A ready-to-use Naukri profile headline tailored specifically for this job.
  User can copy directly into Naukri before applying.

SECTION 8 — SMART APPLY (JOB AUTOMATION ENGINE)
--------------------------------------------------------------------------------

Page: 5_Smart_Apply.py (configuration + results dashboard)
Runner: job_applier/run.py (executes locally on user's Windows PC)
Scheduler: job_applier/watcher.py (background process)

This is the most complex module. The web app configures preferences and
shows results. The actual browser automation runs on the user's local PC
(required because Naukri blocks data centre IPs — residential IP is needed).

8.1 WHAT THE WEB APP DOES (5_Smart_Apply.py)

  Tab 1 — Configure & Download:
    User sets:
      - Job search keywords (auto-populated from resume target_title + alt_titles)
      - Preferred locations (multi-select: Bengaluru, Pune, Mumbai, Hyderabad,
        Chennai, Pune, Navi Mumbai, Remote, and 7 other Indian cities)
      - Current location (used to fill Naukri's Easy Apply modal)
      - Minimum salary in LPA
      - Maximum jobs to apply per run (1-10 slider, default 5)
      - Minimum years of experience filter
      - Domains to avoid (multi-select: Capital Markets/Investment Banking,
        Pharma/Clinical, K-12 Education, FMCG Brand Management, and others)
      - Alternate job titles (other titles to search for besides target title)

    Output: A config.zip downloaded to the user's PC containing:
      - config.json with all preferences + encrypted Naukri password +
        Fernet key + Anthropic API key + Supabase keys
      - run.py (the automation script)
      - requirements.txt
      - install instructions

  Tab 2 — Run History:
    Shows last 30 automation runs fetched from Supabase (or local file fallback).
    Each run shows: date, jobs found, jobs matched, jobs applied, jobs skipped,
    applied list (title + company), skipped list (title + company + reason),
    HR invites detected + whether applied.

  Tab 3 — Recruiter Inbox:
    Shows last 50 HR invites detected across all runs, fetched from Supabase.
    Each invite shows: company, title, HR name, date detected, apply status.

8.2 LOCAL RUNNER (run.py) — FULL AUTOMATION FLOW

  Prerequisite: User has installed Python + Playwright on their Windows PC
  and extracted the config.zip.

  STEP 1 — Load Config
    Reads config.json. Decrypts Naukri password using Fernet symmetric
    decryption (key from config.json, encrypted password from config.json).

  STEP 2 — Launch Browser
    Uses Playwright async_playwright with Chromium.
    In headless=True mode (scheduled/service runs): launches real Chrome
    (channel="chrome") to bypass Naukri's bot detection. Falls back to
    Playwright Chromium if real Chrome isn't installed.
    Bot detection evasion:
      - navigator.webdriver property set to undefined via init script
      - --disable-blink-features=AutomationControlled launch arg
      - Realistic user agent string (Chrome 133)
      - Indian locale (en-IN)
      - Standard 1366x768 viewport
    In headless=False mode: visible browser for debugging.

  STEP 3 — Naukri Login
    Navigates to https://www.naukri.com/nlogin/login
    Fills email + password fields
    Clicks Login button
    Waits for successful login (checks for profile menu or login page absence)
    On failure: logs error, run aborts

  STEP 4 — Recruiter Inbox / NVites Check (P1 — Highest Priority)
    Navigates to https://www.naukri.com/mnjuser/inbox
    Scrapes all NVite cards using selector: div.card.inbox-company-card
    From each card extracts:
      - data-mailid attribute (unique identifier)
      - span.title or span.ellipsis.title (job title)
      - span.comp-name (company name, strips "Hiring for " prefix)
      - span.posted-by-txt (recruiter name, strips "Posted by " prefix)
      - span.date-time-wrap (date posted)
    Stores mail_id for later use in applying
    Fallback: if inbox is empty, checks homepage for power-invite-card widgets
      (selector: div.power-invite-card / div.rmj-card)
    Sends instant ntfy push notification for each new HR invite detected
    (title: "HR Invite — {company}", body: role + recruiter + status, priority: urgent)

  STEP 5 — Regular Job Search (P2)
    Constructs search URL from:
      https://www.naukri.com/{keyword-slugified}-jobs?experience={exp_min}&location={locations}
    Scrapes job listing cards from search results page
    Extracts per job: title, company, location, experience, salary, URL,
    whether Easy Apply is available
    Collects up to configured maximum jobs

  STEP 6 — AI Domain Matching (for each job and each invite)
    Claude is called for each candidate job/invite with the question:
    "Does this job match this candidate's domain and target role?"
    Inputs: job title, company, job description snippet, candidate's
    domain_family, target_title, avoid_domains list
    Output: matched (True/False) + reason
    If matched=False: job is added to skipped_list with reason
    This prevents applying to irrelevant roles (e.g., a TPM getting matched
    to a Capital Markets role)
    Note: Claude API 529 overload errors are treated as SKIP (non-fatal)

  STEP 7 — Resume Tailoring (for matched jobs)
    Claude reads the job details and the user's resume data and produces:
      - A tailored professional summary (3-4 sentences)
      - Reordered and tweaked ATS keywords for this specific job
    This tailored version is used if there's a form field for summary
    during the apply flow

  STEP 8 — Applying to Jobs

    8A. NVite Inbox Apply Flow (for HR invites with mail_id):
      - Navigates back to https://www.naukri.com/mnjuser/inbox
      - Finds card by data-mailid attribute
        (fallback: uses first card if exact match not found)
      - Clicks card to open detail panel on right
      - Finds Apply / Apply Now / Easy Apply button in the detail panel
      - Checks for "Applied" text (skip if already applied)
      - Clicks Apply button
      - Calls Easy Apply Modal Handler (see 8C below)
      - Clicks Submit Application / Submit if modal appears
      - Fires Make.com webhook with type="hr_invite_apply" (if configured)

    8B. Regular Job Easy Apply Flow (for jobs with direct URL):
      - Navigates to job URL
      - Scans for Apply button using multiple selectors:
          button.styles_apply-button__N2-qy (Naukri 2024 class)
          button[class*='apply-button']
          a[class*='apply-button']
          .applyButton
          button.apply-button
          button:has-text('Easy Apply')
          button:has-text('Apply Now')
          button:has-text('Apply')
          #apply-button
      - Checks if button shows "Applied" or is disabled (skip if so)
      - Clicks Apply button
      - Calls Easy Apply Modal Handler (see 8C below)
      - Clicks Submit Application / Submit if a modal appeared
      - Fires Make.com webhook with type="easy_apply" (if configured)

    8C. Easy Apply Modal Handler (_handle_easy_apply_modal):
      After clicking Apply, a modal may appear asking for profile details.
      The handler tries to fill:

      Current Location field:
        Tries multiple selectors for the city typeahead input:
          input[placeholder*='current city' i]
          input[id*='currentCity' i]
          input[id*='currentLocation' i]
          [class*='currentCity'] input
          [class*='location'] input[type='text']
        If found: triple-clicks (selects all), types current_location from config
        Waits for typeahead dropdown to appear
        Tries to click first suggestion from the dropdown:
          [class*='suggestItem'], [class*='suggestion-item'],
          [class*='autocomplete'] li, [role='option'], etc.
        If no dropdown: presses Enter to confirm

      Notice Period field:
        Tries selectors: select[id*='notice' i], [class*='noticePeriod'] select
        If found: selects index=1 (typically "Immediate" or "0-15 days")

      The handler is intentionally non-blocking — if any field is not found
      or fails, it logs and continues. Does not abort the apply flow.

  STEP 9 — Human-like Delays
    Between each job: random.uniform(2, 4) second delay
    After each click: random.randint(1500, 2500) ms wait
    Page loads: random.randint(2000, 3000) ms wait
    This prevents rate-limiting and looks human to Naukri's bot detection

  STEP 10 — Results Sync to Supabase
    After run completes, _sync_results() posts to Supabase:

    run_history table row:
      - user_email, run_date, jobs_found, jobs_applied, jobs_skipped
      - applied_list (array of {title, company} objects)
      - skipped_list (array of {title, company, reason} objects)
      - hr_invites (array of invite objects)
      - errors (array of error messages)

    hr_invites table rows (one per invite):
      - user_email, company, title, url, hr_name, applied (bool),
        reason, detected_at

    Fallback modes if Supabase not configured:
      Mode 2: legacy api_ingest URL (blocked by Streamlit OAuth gate — not used)
      Mode 3: direct file write to data/users/{user_id}/ on the same machine

  STEP 11 — Run Complete Notification
    Sends ntfy push: "CareerOS Run Complete"
    Body: "HR invites processed: X | Jobs scanned: Y | Applied: Z"
    Priority: low (doesn't buzz the phone, shows as silent notification)

8.3 WATCHER / SCHEDULER (watcher.py)

  A persistent background process (runs on user's PC, stays alive).

  Three trigger mechanisms:

    A. Scheduled Times (default: 9:30 AM and 2:00 PM daily)
       Checks current time every 60 seconds.
       Each scheduled slot runs once per day (tracked in watcher_state.json).
       Prevents double-running if watcher is restarted mid-day.

    B. Make.com Webhook Trigger
       Runs a local HTTP server on port 8765.
       Endpoint: POST /trigger
       Creates a trigger.txt file that the main loop detects on next tick.
       Use case: trigger a run from a WhatsApp message via Make.com + ngrok.
       For external access: user runs ngrok to expose port 8765 publicly.

    C. Manual Trigger File
       Creating any file named trigger.txt in the job_applier directory
       triggers an immediate run on the next 60-second check.

  Run timeout: 7200 seconds (2 hours) — prevents hung runs.
  All watcher events logged to job_applier/logs/watcher.log.

  Windows service installation: install_service.ps1 PowerShell script
  for setting up the watcher as a Windows startup service.

SECTION 9 — NOTIFICATION ENGINE
--------------------------------------------------------------------------------

Service used: ntfy.sh (free, open source, no account required)
Module: modules/notifications/ntfy.py

9.1 TOPIC DERIVATION
  Each user gets a unique, non-guessable topic: "careeros-{md5_hash[:10]}"
  Derived from their email. Deterministic — same email always gets same topic.
  Users subscribe in the ntfy mobile app using this topic code.

9.2 NOTIFICATION TYPES

  HR Invite Detected:
    Priority: URGENT (full screen takeover on phone)
    Tags: fire, briefcase
    Title: "HR Invite — {Company Name}"
    Body: Role + HR Name + Status (Auto-applied / Needs your attention)
    Click action: Link to the job/invite URL

  Profile Viewed (designed, not yet wired to Naukri profile view events):
    Priority: HIGH
    Tags: eyes
    Title: "Profile Viewed — {Company Name}"
    Body: "An HR from this company viewed your Naukri profile. Invite may follow."

  Run Complete:
    Priority: LOW (silent notification)
    Tags: white_check_mark
    Title: "CareerOS Run Complete"
    Body: HR invites processed + Jobs scanned + Applied counts

  Test Notification (from Setup page):
    Priority: DEFAULT
    Tags: tada
    Title: "CareerOS is connected!"
    Body: Confirmation message

9.3 HOW NTFY WORKS
  All notifications POST to https://ntfy.sh/{topic} with HTTP headers.
  No API key needed for the default tier. Free up to reasonable limits.
  Works on Android, iOS, Web, and desktop.
  Open source — can be self-hosted if needed in future.

    hr_invites.json       — last 50 HR invites detected (local fallback)
    cover_letters.json    — last 10 generated cover letters

10.2 SUPABASE (primary for automation sync)

  Supabase project: jberxpkuyezqbveztnep.supabase.co
  Authentication: publishable API key (row-level access, no admin)

  Tables:
    run_history:
      id, created_at, user_email, run_date, jobs_found, jobs_applied,
      jobs_skipped, applied_list (jsonb), skipped_list (jsonb),
      hr_invites (jsonb), errors (jsonb)

    hr_invites:
      id, created_at, user_email, company, title, url, hr_name,
      applied (boolean), reason, detected_at

  Write path: local run.py → Supabase REST API (POST /rest/v1/{table})
  Read path: web app 5_Smart_Apply.py → Supabase REST API (GET /rest/v1/{table})

  Fallback: if Supabase is unreachable or unconfigured, reads fall back to
  local JSON files (apply_history.json and hr_invites.json).

  Why Supabase instead of direct API call to Streamlit web app:
    Streamlit Cloud's server-side rendering puts Google OAuth in front of
    every route including API endpoints. The local runner cannot pass Google
    OAuth tokens. Supabase bypasses this entirely with a direct REST call
    using the publishable key.

10.3 STREAMLIT SESSION STATE

  In-session (not persisted to disk):
    - rb_step, rb_session, rb_chat, rb_resume, rb_doc_bytes, rb_exchange
      (Resume Builder conversation state for the current browser session)
    - last_cover_letter (most recently generated cover letter)

  Session state is reset on browser refresh. Long-running resume builder
  sessions are persisted to resume_chat.json via store.save_chat_history()
  after each exchange so they can be restored on next visit.

SECTION 13 — WHAT IS NOT YET BUILT (HONEST GAPS)
--------------------------------------------------------------------------------

These are capabilities referenced in the UI as "Coming Soon" or identified
as significant missing pieces for the automation engine:

  1. COMPANY PORTAL APPLY (BIGGEST GAP)
     When a Naukri job shows "Apply on Company Site" instead of Easy Apply,
     the bot follows the redirect to the company's own ATS portal and does
     nothing. It returns False and skips the job.

     This affects approximately 40-60% of mid-to-senior PM/TPM roles on
     Naukri, since most large companies use their own portals:
       - Workday (most common for large Indian enterprises — Tata, Infosys, etc.)
       - Greenhouse (startups and product companies)
       - Lever (mid-size tech companies)
       - iCIMS (some large MNCs)
       - SAP SuccessFactors (manufacturing, BFSI)

     Each ATS has a completely different form structure. Not yet built.

  2. APPLICATION TRACKER
     Shown on dashboard as "Coming Soon".
     No tracking of: applied-to job status, interview scheduling,
     follow-up reminders, response rate analytics.

  3. INTERVIEW PREP MODULE
     Shown on dashboard as "Coming Soon".
     No interview Q&A generation, HR round prep, STAR-format answers,
     company-specific question banks.

  4. PROFILE VIEW MONITORING
     notify_profile_view() function exists in ntfy.py but is not wired
     to any Naukri scraping. Naukri shows profile views in
     "Who viewed my profile" section. Not yet automated.

  5. LINKEDIN JOB APPLICATIONS
     The automation engine only targets Naukri. LinkedIn Easy Apply
     is not built. LinkedIn jobs are a major parallel channel.

  6. RESUME AUTO-UPDATE ON NAUKRI
     CareerOS generates a Word resume and profile optimizer content but
     does not automatically update the Naukri profile (headline, summary,
     skills, employment descriptions). User must copy-paste manually.

  7. MULTI-USER / TEAMS
     Designed as a single-tenant SaaS. No admin dashboard, no user
     management, no billing/subscription engine.

  8. INTERVIEW SCHEDULING INTEGRATION
     No calendar integration (Google Calendar, Outlook) for interview
     slot management or automatic scheduling.

  9. WHATSAPP TRIGGER (PARTIAL)
     The watcher.py webhook exists and can receive Make.com POST requests.
     The Make.com scenario to trigger a run from a WhatsApp message is
     designed but depends on external Make.com configuration. Not
     bundled or automated end-to-end.

  10. NAUKRI RESUME UPDATE AUTOMATION
      After applying, CareerOS does not refresh the "resume freshness"
      signal on Naukri (which affects search ranking). Old resumes rank
      lower — manual refresh or automation needed.

SECTION 14 — USER JOURNEY END TO END
--------------------------------------------------------------------------------

Day 1 — First time user:
  1. Visits app URL → Google sign-in
  2. Setup page: enters Naukri credentials, chooses preferred locations,
     sets up ntfy on phone, clicks "Test Notification", saves.
  3. Role Clarity page (optional): answers 5 questions,
     gets role recommendation + salary band.
  4. Resume Builder: 10-15 minute chat in Hindi or English,
     downloads Word resume.
  5. Profile Optimizer: generates Naukri headline/summary/skills,
     copy-pastes into Naukri profile manually.
  6. ATS Checker: pastes a target job description, sees keyword gaps,
     updates resume if needed.
  7. Cover Letter: generates a tailored letter for their top application.

Day 2+ — Automation running:
  1. User downloads config.zip from Smart Apply page.
  2. Extracts on Windows PC, runs: python watcher.py
  3. At 9:30 AM and 2:00 PM every day:
     - Bot logs into Naukri automatically
     - Checks NVites inbox for fresh HR invites
     - Searches for matching jobs
     - Claude scores each job against the profile
     - Applies to matching jobs (Easy Apply only)
     - Syncs results to Supabase
     - Sends ntfy push notification summary
  4. On phone: instant alert if an HR invite arrives (urgent priority push).
  5. On web app: Run History tab shows all applications.
  6. User can check Smart Apply → Recruiter Inbox for all HR invite history.
